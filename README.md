# Wallet Service

Высоконагруженный REST-сервис для управления электронными кошельками. Приложение поддерживает операции пополнения и снятия средств с гарантией атомарности транзакций и отсутствием потерь данных при высокой конкурентной нагрузке (до 1000+ RPS на один кошелек).

## 1. Описание проекта

Проект реализует базовую логику процессинга финансовых операций. Ключевым требованием при разработке было обеспечение отказоустойчивости и консистентности данных в условиях жесткой конкуренции за ресурсы (множество одновременных запросов на изменение баланса одного и того же кошелька).

Приложение упаковано в Docker-контейнеры, использует миграции базы данных для версионирования схемы и покрывается интеграционными тестами с поднятием изолированного окружения через Testcontainers.

Ключевые возможности:
- Атомарные финансовые транзакции без использования блокировок на уровне приложения.
- Защита от гонок данных (Race Conditions) и взаимных блокировок (Deadlocks).
- Строгая валидация входящих данных и кастомная обработка ошибок (400, 404, 409).
- Полная контейнеризация (Docker + Docker Compose) для быстрого развертывания.
- Управление миграциями базы данных через Liquibase.
- Интеграционные тесты с использованием Testcontainers.
- Конфигурация через переменные окружения без необходимости пересборки образов.

## 3. Архитектура и решение проблемы конкурентного доступа

Главный архитектурный вызов данного проекта — обеспечение корректной работы при нагрузке в 1000 запросов в секунду (RPS) на одну и ту же строку в базе данных.

### Почему не подошли стандартные подходы

1. **Пессимистичные блокировки (SELECT ... FOR UPDATE)**: При 1000 RPS транзакции выстраиваются в огромную очередь. Это приводит к таймаутам блокировок, исчерпанию пула соединений HikariCP и, как следствие, к ошибкам 500 (Internal Server Error).
2. **Оптимистичные блокировки (@Version)**: При высокой конкуренции постоянно возникает исключение `OptimisticLockingFailureException`. Для его обработки требуется сложная логика ретраев (повторных попыток) на стороне Java, что также ведет к перегрузке и ошибкам 5xx.

### Примененное решение

Использование атомарного обновления на уровне СУБД одним SQL-запросом через Spring Data JPA `@Query`:

```sql
UPDATE wallet 
SET balance = balance + :amount, 
    version = version + 1 
WHERE id = :id 
  AND (balance + :amount) >= 0
```

### Преимущества данного подхода

- Транзакция удерживает эксклюзивную блокировку строки (Row-level lock) минимально возможное время (доли миллисекунды).
- Проверка достаточности средств выполняется атомарно внутри движка PostgreSQL.
- Полностью исключены гонки данных (Race Conditions) и взаимные блокировки (Deadlocks).
- Возвращаемое значение (количество обновленных строк: 1 или 0) позволяет приложению точно и быстро различить ошибки "Кошелек не найден" (404) и "Недостаточно средств" (409) без дополнительных SELECT-запросов.

## 4. API Документация

### 4.1 Операция над кошельком (Пополнение / Снятие)

**Эндпоинт:** `POST /api/v1/wallet`

**Тело запроса (JSON):**

```json
{
  "walletId": "a0eebc99-9c0b-4ef8-bb6d-6bb9bd380a11",
  "operationType": "DEPOSIT",
  "amount": 1000.50
}
```

**Параметры:**

- `walletId` (UUID, обязательно) — Идентификатор кошелька.
- `operationType` (Enum, обязательно) — Тип операции: `DEPOSIT` (пополнение) или `WITHDRAW` (снятие).
- `amount` (Decimal, обязательно) — Сумма операции. Должна быть строго больше 0.

**Пример запроса (cURL):**

```bash
curl -X POST http://localhost:8080/api/v1/wallet \
  -H "Content-Type: application/json" \
  -d '{
    "walletId": "a0eebc99-9c0b-4ef8-bb6d-6bb9bd380a11",
    "operationType": "DEPOSIT",
    "amount": 500
  }'
```

### 4.2 Получение баланса кошелька

**Эндпоинт:** `GET /api/v1/wallets/{WALLET_UUID}`

**Пример запроса (cURL):**

```bash
curl http://localhost:8080/api/v1/wallets/a0eebc99-9c0b-4ef8-bb6d-6bb9bd380a11
```

**Пример успешного ответа (200 OK):**

```json
{
  "id": "a0eebc99-9c0b-4ef8-bb6d-6bb9bd380a11",
  "balance": 1500.0000,
  "version": 2
}
```

### 4.3 Коды ответов и обработка ошибок

Приложение возвращает стандартизированный JSON-формат для всех ошибок через глобальный обработчик `@RestControllerAdvice`.

| HTTP Статус | Условие возникновения | Пример сообщения |
|---|---|---|
| 200 OK | Операция успешно выполнена / Баланс получен | (Тело ответа или пусто) |
| 400 Bad Request | Невалидный JSON, некорректный UUID, сумма <= 0 | "Некорректный формат walletId: ожидается UUID" |
| 404 Not Found | Кошелек с указанным UUID не существует | "Кошелек не найден: {uuid}" |
| 409 Conflict | Недостаточно средств для операции WITHDRAW | "Недостаточно средств на кошельке: {uuid}" |
| 500 Internal Server Error | Непредвиденная ошибка сервера | "Внутренняя ошибка сервера" |

## 5. Схема базы данных

Таблица `wallet` создается и управляется с помощью миграций Liquibase (`src/main/resources/db/changelog/db.changelog-master.xml`).

| Колонка | Тип данных | Ограничения | Описание |
|---|---|---|---|
| `id` | UUID | PRIMARY KEY | Уникальный идентификатор кошелька |
| `balance` | NUMERIC(19, 4) | NOT NULL | Текущий баланс (используется масштаб 4 для точности) |
| `version` | BIGINT | NOT NULL, DEFAULT 0 | Версия записи (для аудита и оптимистичных блокировок) |

## 6. Запуск и развертывание

### 6.1 Требования

- Установленный Docker и Docker Compose.

### 6.2 Шаги для запуска

1. Склонируйте репозиторий и перейдите в корневую папку:

   ```bash
   git clone <repository-url>
   cd wallet-service
   ```

2. Выполните команду для сборки и запуска (база данных и приложение поднимутся автоматически, Liquibase накатит миграции):

   ```bash
   docker-compose up --build
   ```

3. Дождитесь сообщения `Started WalletServiceApplication` в логах контейнера `wallet_app`.
4. Приложение будет доступно по адресу: `http://localhost:8080`
5. PostgreSQL будет доступен на порту `5432` (localhost).

### 6.3 Остановка и очистка данных

Остановка с сохранением данных в БД:

```bash
docker-compose down
```

Полная остановка с удалением тома базы данных (сброс всех кошельков):

```bash
docker-compose down -v
```

## 7. Конфигурация

Все параметры конфигурируются через переменные окружения без необходимости пересборки Docker-образов. Вы можете передать их через `docker-compose.yml`, через `.env` файл или через командную строку.

### 7.1 Доступные переменные окружения

| Переменная | По умолчанию | Описание |
|---|---|---|
| `POSTGRES_DB` | `wallet_db` | Имя базы данных |
| `POSTGRES_USER` | `wallet_user` | Пользователь БД |
| `POSTGRES_PASSWORD` | `wallet_password` | Пароль пользователя БД |
| `POSTGRES_PORT` | `5432` | Внешний порт PostgreSQL |
| `SERVER_PORT` | `8080` | Порт приложения |
| `HIKARI_POOL_SIZE` | `20` | Размер пула соединений |
| `JPA_DDL_AUTO` | `validate` | Режим Hibernate DDL |
| `LIQUIBASE_ENABLED` | `true` | Включение/отключение Liquibase |

### 7.2 Пример запуска с кастомными параметрами

Создайте файл `.env` в корне проекта:

```env
POSTGRES_PASSWORD=secure_password
HIKARI_POOL_SIZE=50
SERVER_PORT=8081
```

Docker Compose автоматически подхватит эти значения при следующем `docker-compose up`.

### 7.3 Запуск с передачей переменных через CLI

**Linux / macOS:**

```bash
POSTGRES_PASSWORD=secure_pass HIKARI_POOL_SIZE=50 docker-compose up
```

**Windows PowerShell:**

```powershell
$env:POSTGRES_PASSWORD="secure_pass"
$env:HIKARI_POOL_SIZE="50"
docker-compose up
```

## 8. Тестирование

### 8.1 Интеграционные тесты

Проект содержит интеграционные тесты (`WalletIntegrationTest`), которые используют Testcontainers для поднятия временной базы данных PostgreSQL. Это гарантирует, что тесты проверяют реальное взаимодействие с СУБД, а не in-memory заглушки.

Для запуска тестов (требуется локально установленный Maven и Docker):

```bash
mvn clean test
```

Тесты покрывают:
- Успешные операции DEPOSIT и WITHDRAW.
- Ошибку 404 при несуществующем кошельке.
- Ошибку 409 при недостатке средств.
- Ошибку 400 при невалидном UUID и битом JSON.
- Тест на высокую конкуренцию (100 параллельных запросов на списание средств с проверкой итогового баланса).

### 8.2 Ручное нагрузочное тестирование

1. Создайте тестовый кошелек через psql:

   ```bash
   docker exec -it wallet_db psql -U wallet_user -d wallet_db \
     -c "INSERT INTO wallet (id, balance, version) VALUES ('a0eebc99-9c0b-4ef8-bb6d-6bb9bd380a11', 1000.0000, 0);"
   ```

2. Используйте утилиты вроде `Apache JMeter`, `k6` или `wrk` для отправки тысяч параллельных POST-запросов на `/api/v1/wallet`.

3. Убедитесь, что итоговый баланс строго равен математическому ожиданию, а в логах отсутствуют ошибки 5xx.

## 10. Устранение неполадок

### 10.1 Контейнер `wallet_app` постоянно перезапускается

**Симптом:** В логах ошибка подключения к БД или Liquibase.

**Решение:** Убедитесь, что база данных полностью инициализирована. В `docker-compose.yml` настроен `healthcheck` для PostgreSQL, и приложение ждет статуса `healthy`. Если проблема сохраняется, выполните `docker-compose down -v` и запустите заново.

### 10.2 Порт 8080 или 5432 уже занят

**Симптом:** Ошибка `port is already allocated` при запуске `docker-compose up`.

**Решение:** Измените маппинг портов в `docker-compose.yml` или создайте `.env` файл с переменными `SERVER_PORT=8081` и `POSTGRES_PORT=5433`.

### 10.3 Liquibase не применяет миграции

**Симптом:** Таблица `wallet` не создается, приложение падает с ошибкой `Table 'wallet' doesn't exist`.

**Решение:** Убедитесь, что файл `db.changelog-master.xml` находится строго по пути `src/main/resources/db/changelog/` и в `application.yml` указан корректный `change-log: classpath:db/changelog/db.changelog-master.xml`.

### 10.4 Ошибки 500 при невалидном UUID

**Симптом:** Приложение возвращает 500 вместо 400 при передаче некорректного `walletId`.

**Решение:** Убедитесь, что в `GlobalExceptionHandler.java` добавлены обработчики `HttpMessageNotReadableException` и `MethodArgumentTypeMismatchException`. Обратите внимание на правильный импорт: `org.springframework.web.method.annotation.MethodArgumentTypeMismatchException`.

### 10.5 Проблема с экранированием JSON в PowerShell

**Симптом:** cURL возвращает ошибку `JSON parse error`.

**Решение:** PowerShell 5.1 некорректно обрабатывает кавычки. Используйте встроенный `Invoke-RestMethod` или сохраняйте JSON в файл и передавайте через `@filename`:

```powershell
Invoke-RestMethod -Uri http://localhost:8080/api/v1/wallet `
  -Method Post `
  -ContentType "application/json" `
  -Body '{"walletId": "...", "operationType": "DEPOSIT", "amount": 100}'
```

### 10.6 Отладка slim-контейнеров через Docker Debug

**Симптом:** Необходимо проинспектировать файловую систему контейнера или изменить конфигурационные файлы, но образ не содержит shell (например, используется `eclipse-temurin:17-jre-alpine`).

**Решение:** Используйте команду `docker debug`, которая позволяет получить shell-доступ в любой контейнер или образ, даже если они не содержат shell. Docker Debug предоставляет собственный toolbox с предустановленными инструментами (`vim`, `nano`, `htop`, `curl`), не модифицируя при этом сам образ.

Пример использования:

```bash
# Получить shell в запущенном контейнере
docker debug wallet_app

# Внутри debug shell можно выполнять любые команды
docker > ls /app
docker > vim /app/BOOT-INF/classes/application.yml
docker > exit
```

Установка дополнительных инструментов внутри debug shell:

```bash
docker > install nmap
docker > nmap --version
```

Преимущества подхода:
- Образ остается slim и безопасным (без bash и лишних утилит).
- Все изменения в debug shell не влияют на работающий контейнер.
- Возможность удаленной отладки через флаг `--host`:

```bash
docker debug --host ssh://user@remote-host wallet_app
```
